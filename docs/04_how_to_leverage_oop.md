# オブジェクト指向を活かした高度なプログラミング手法

オブジェクト指向（OOP）のメリットを最大限に引き出し、「スパゲッティコード（絡み合って修正不能なコード）」を防ぐための具体的な設計手法（デザインパターンと原則）です。

## 1. 原則：継承よりコンポジション（Composition over Inheritance）

OOPの罠としてよく挙げられるのが「深い継承ツリー」です。
例えば、「移動できる敵」と「空を飛ぶ敵」と「魔法を使う敵」と「空を飛んで魔法を使う敵」を作ろうとすると、継承関係が破綻します。

現代のOOPでは、**コンポーネント（部品）の組み合わせ（コンポジション）**でキャラクターを作ることが推奨されます。
*   クラスに直接ロジックを書くのではなく、`MovementComponent`（移動部品）や `EnglishQuizComponent`（英語出題部品）という別クラスを作ります。
*   敵キャラクターを生成する際、必要な部品をインスタンスの配列として持たせます。これにより、レゴブロックのように自由に機能の組み合わせが可能になります。

## 2. ゲーム開発で必須のGoFデザインパターン

GoF（Gang of Four）デザインパターンのうち、ゲーム開発で特に威力を発揮するものを採用します。

### A. State（状態）パターン
*   **課題**: プレイヤーの `update()` メソッドの中が `if (isPlayingAnimation) { ... } else if (isJumping) { ... } else if (isAnsweringQuiz) { ... }` という巨大なif文の嵐になる。
*   **解決法**: 「待機状態クラス」「移動状態クラス」「クイズ回答中状態クラス」を個別に作成し、共通の `IState` インターフェースを持たせます。プレイヤーは現在のアクティブな状態クラスに処理を丸投げ（委譲）します。これにより、バグのないスムーズなアクション遷移が実装可能になります。

### B. Observer（観察者）パターン
*   **課題**: 敵を倒したとき、「スコアシステムに得点を加算」「UIシステムに文字を表示」「オーディオシステムに音を鳴らす」といった処理を書くと、敵クラスが他のシステムに密結合（依存）してしまい、コードが汚くなります。
*   **解決法**: イベント発行（Pub/Sub）モデルを使います。敵が倒れた時は、ただ空に向かって「敵が倒れたよ！」というイベントを発行（Emit）するだけにします。スコア管理やUI管理のクラスは、常にそのイベントを「監視（Observe）」しておき、イベントが飛んできたら勝手に自分の処理を行います。これによってクラス間の依存がなくなり、後から実績（トロフィー）機能などを追加するのが非常に簡単になります。

### C. Command（コマンド）パターン
*   **課題**: プレイヤーの入力（キーボードやスマホのタップ）を直接キャラクターの移動コードに結びつけると、キーコンフィグ（ボタン割り当て変更）の機能を作るのが非常に困難になります。
*   **解決法**: 入力を「MoveUpCommand」などの「コマンドオブジェクト」に変換してからキャラクターに渡します。これを利用すると、キーコンフィグが容易になるだけでなく、入力履歴を保存して「リプレイ機能」を作ったり、「戻る（Undo）」機能を実装することが可能になります。

## 3. SOLID原則（特に重要な2つ）

*   **単一責任の原則 (Single Responsibility Principle)**:
    1つのクラスが持つ責任（変わる理由）は常に1つであるべきです。例えば、「英語の単語データ」を管理するクラスと、「それを画面に描画する」クラスは完全に分けるべきです。
*   **依存性逆転の原則 (Dependency Inversion Principle)**:
    具体的なクラス（例: `StripePayment`クラス）に依存するのではなく、抽象（例: `IPaymentProcessor` インターフェース）に依存させることで、後から決済システムを別のものに変えたくなった時でも、ゲーム側のコードを一切書き換えずに済むようにします。
